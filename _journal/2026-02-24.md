### Key insight: tool call streaming gaps

When the agent uses tools, the flow is:
```
text streams → model calls tool → SILENCE (tool executes) → tool result → text resumes
```
Each tool call is a separate LLM API round-trip. The silence can last 5-30 seconds depending on the tool. The tool indicator now makes this visible to the user.

### Root cause: job timeout mystery solved

The `$timeout = 600` on `ProcessChatMessage` was being ignored because:
1. Queue worker had no `--timeout` flag → defaults to 60s (kills via `pcntl_alarm()`)
2. `retry_after=90` → queue re-dispatches "stuck" jobs after 90s
3. `--tries=3` → three attempts, each killed early, appearing as ~300s total timeout

Fix: `--timeout=660` on worker + `retry_after=900` in config + `--tries=1`.

### Tool indicator investigation

Confirmed the laravel/ai SDK **does** broadcast `tool_call` and `tool_result` stream events through the full pipeline:
- Prism's Anthropic stream handler emits `ToolCallEvent` / `ToolResultEvent`
- `PrismStreamEvent::toLaravelStreamEvent()` translates to Laravel `ToolCall` / `ToolResult`
- `Promptable::broadcastNow()` iterates all stream events and broadcasts via Reverb
- Frontend `.tool_call` / `.tool_result` listeners should match

The indicator code is correct — it only appears when the model actually calls tools (HttpRequest, Bash, CurrentDateTime). Pure text responses won't trigger it.

### gw.goldcoast.org Postfix fix

**Problem:** Thunderbird sending via `gw.goldcoast.org:465` timed out with "connection timed out".

**Root cause:** When the IPv6 `[fdaa:5e90:abf::1]:465` listener was added to `/etc/postfix/master.cf`, the `-o` overrides (including `smtpd_tls_wrappermode=yes`) shifted from the IPv4 `192.168.1.1:465` entry to the IPv6 entry. Postfix `master.cf` attaches `-o` options to the immediately preceding `inet` line — so the IPv4 listener silently reverted to plain SMTP while only IPv6 got implicit TLS.

Thunderbird expects implicit TLS on port 465 — the plaintext SMTP banner caused the TLS handshake to fail, appearing as a timeout.

**Fix:** Duplicated all `-o` overrides under both the IPv4 and IPv6 listener entries in `master.cf`, then `postfix reload`. Verified with `openssl s_client -connect 192.168.1.1:465` — Let's Encrypt cert now served correctly on both.

| Cert | SANs | Reload command |
|------|------|---------------|
| `motd.com` | motd.com, www, **app.motd.com**, autoconfig, autodiscover | `chmod 644 privkey.pem && systemctl restart frankenphp` |
| `mail.motd.com` | mail.motd.com | `chmod 644 privkey.pem && systemctl restart stalwart-mail frankenphp` |

## Session 6: Mesh Architecture Discovery — Topology, DNS, and Markdown Routing

### Mesh topology

Explored fault tolerance without full O(N²) mesh. A **2-connected (biconnected)** graph — minimum degree 2 with no cut vertices — survives any single node failure. **Ring + random chords** is the optimal topology: O(N) connections, O(log N) propagation, high multi-failure survivability. At 3 nodes it's academic (triangle = full mesh); matters at 8+.

### DNS primary/secondary as state replication

The DNS model maps directly to markweb: primary (mko) holds authoritative state, secondaries pull via AXFR, promote if primary dies. No Raft, no distributed DB — just "pull latest snapshot, serve if primary is gone." 40 years of battle-tested infrastructure.

### Naming: mesh, not cluster

A **cluster works as one**. A **mesh works together but independently**. markweb nodes are self-sufficient (own DB, mail, embeddings, AI) — they report in but don't need each other to operate. Individual participants are **nodes** (not leaves — every node can become primary).

### WebSocket vs DNS for discovery

Both have a role at different layers:
- **Reverb (WebSocket)** = control plane — real-time, rich metadata, but only Laravel speaks it
- **PowerDNS (UDP)** = data plane — universal, Stalwart/curl/systemd all speak it

The hybrid: Reverb carries mesh state changes → listener updates PowerDNS records → AXFR propagates to all nodes → every service discovers peers via standard DNS. This is how Consul works.

### DNS record types for the mesh

| Priority | Record | Value |
|----------|--------|-------|
| 1 | **SRV** | Service discovery with failover/load balancing. Thunderbird already queries these. |
| 2 | **DNS-SD** (PTR+SRV+TXT) | Zero-conf — new node discovers all mesh services with one query |
| 3 | **TLSA/DANE** | Stalwart has built-in DANE. Mesh nodes verify certs via DNS, no CA needed. |
| 4 | **SSHFP** | SSH host key fingerprints — eliminates TOFU warnings between nodes |
| 5 | **SVCB/HTTPS** | HTTP/3 advertisement for FrankenPHP |
| 6 | **OPENPGPKEY** | PGP key distribution — low value at 3 nodes |
| 7 | **LOC** | Geographic location — revisit if mesh goes multi-region |

### Mesh-local DNS — private service routing

Every node gets a local resolver. Mesh domains resolve to WG IPs:
- `kanary.org` MX → 172.16.1.210 (mko, over WG)
- `motd.com` MX → 172.16.1.9 (mmc, over WG)
- External domains → public DNS (fallthrough)

Inter-node mail never leaves WireGuard. Eliminates `/etc/hosts` fragility (cloud-init wipes on mmc). Three implementation options: dnsmasq (lightest), Unbound (mid), PowerDNS Recursor + Auth (full — zone transfers come free).

### Markdown as primary payload

Markdown text exchange is markweb's core purpose. DNS supports this:
- `_markdown._tcp` SRV records route content to the best available node (priority/weight failover)
- TXT records advertise capabilities: `format=commonmark`, `streaming=reverb`, `models=claude-sonnet-4`
- Ollama embedding requests load-balance across nodes via SRV weights
- Inter-node email carrying markdown stays entirely within WG via mesh MX + DANE verification

### goldcoast.org as prototype

Split-horizon DNS is already working: ns1gc (public WAN) + gw2 dnsmasq (private LAN). The mesh adds a third layer (WG) with automatic propagation via zone transfers. The pattern is proven — it just needs generalising.

## Session 7: Intent Engineering Applied to markweb

### What is intent engineering?

The evolution beyond prompt engineering → context engineering → **intent engineering**. Core principle: "Intent is what determines how an agent acts when instructions run out." Declare WHAT outcome you want with constraints and success criteria; the system figures out HOW.

Seven-component framework: Objective, Desired Outcomes, Health Metrics, Strategic Context, Constraints, Decision Authority, Stop Rules.

### Three layers in markweb

**1. Agent layer** — Current `IntentRouter` (`app/Services/Agent/IntentRouter.php`) classifies input TYPE (Command/Query/Task via heuristics), not system OBJECTIVES. Intent engineering would add: objectives ("help user manage communications efficiently"), constraints ("never send email without confirmation"), decision authority ("can search mail autonomously, must escalate bulk operations"), stop rules ("ask rather than guess when confidence is low").

**2. Mesh layer** — The mesh DNS architecture designed in Session 6 IS intent engineering applied to infrastructure:

| Framework Component | Mesh DNS Equivalent |
|-------------------|-------------------|
| Objective | "Mail between nodes stays private on WG" |
| Decision authority | SRV priorities — which node SHOULD handle a service |
| Health metrics | Heartbeats — node liveness every 30s |
| Constraints | DANE/TLSA — "verify connections via this certificate" |
| Strategic context | TXT records — capabilities, versions, models |
| Stop rules | Offline detection — mark stale after 90s |

SRV records are declarative intent: "I intend for JMAP to be served by mko (priority 10) with mmc as fallback (priority 20)." The mesh resolves the how.

**3. Markdown exchange layer** — The most natural fit. Imperative: "POST this markdown to mko at /api/content via WG IP 172.16.1.210." Intent-driven: "I want this markdown to reach kanary.org, encrypted, within 30 seconds." The mesh checks SRV, picks the best node, verifies via DANE, delivers over WG. Node down? Try fallback. WG down? Queue for later.

### Key insight

Every architecture decision in today's sessions maps to intent engineering:
- Heartbeats = health metrics (liveness intent)
- SRV priorities = decision authority (routing intent)
- DANE/TLSA = constraints (trust intent)
- TXT capabilities = strategic context (capability intent)
- DNS-SD = discovery intent (mesh discovers without being told how)
- Ring + chords = resilience intent (survive any single node failure)
- `web.*` convention = naming intent ("this IS the entry point")
