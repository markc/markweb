### Key insight: tool call streaming gaps

When the agent uses tools, the flow is:
```
text streams → model calls tool → SILENCE (tool executes) → tool result → text resumes
```
Each tool call is a separate LLM API round-trip. The silence can last 5-30 seconds depending on the tool. The tool indicator now makes this visible to the user.

### Gotchas discovered

- `env()` returns null after `config:cache` — never call it outside config files
- Laravel's default broadcasting config uses public-facing `REVERB_HOST`/`REVERB_PORT` for both frontend AND server-side Pusher client — breaks with reverse proxy (Caddy)
- Reverb reads app credentials at boot — restarting Reverb required after changing `REVERB_APP_ID`
- Queue workers from other apps (laraclaw/laramail) don't process markweb's database queue
- `dispatchSync()` is simpler but times out on slow models (Opus 4.6) — async queue required
- Reverb default `max_message_size` is 10KB — far too small for LLM streaming with tool results
- Systemd services for queue/reverb need restarting after config changes — `queue:restart` signal unreliable for stale workers

## Session 3: Queue Timeout Fix and Docs Collection

### What changed

Fixed the real cause of job timeouts (queue worker `--timeout` and `retry_after` misconfiguration), rescued Opus HTML artifact, and grew `/docs/` collection to 3 entries.

**3 commits:** `77f7152..73d2ce7`

1. **`77f7152` — Bump queue retry_after to 900s for LLM job timeouts**
   - Default `retry_after=90` causes database queue to re-dispatch long-running LLM jobs

   - Bumped to 900s (15 minutes) — must exceed job timeout

- Fixed `markweb-queue.service`: added `--timeout=660` (was unset, defaulting to 60s — the real timeout killer)

### Root cause: job timeout mystery solved

The `$timeout = 600` on `ProcessChatMessage` was being ignored because:
1. Queue worker had no `--timeout` flag → defaults to 60s (kills via `pcntl_alarm()`)
2. `retry_after=90` → queue re-dispatches "stuck" jobs after 90s
3. `--tries=3` → three attempts, each killed early, appearing as ~300s total timeout

Fix: `--timeout=660` on worker + `retry_after=900` in config + `--tries=1`.

### Tool indicator investigation

Confirmed the laravel/ai SDK **does** broadcast `tool_call` and `tool_result` stream events through the full pipeline:
- Prism's Anthropic stream handler emits `ToolCallEvent` / `ToolResultEvent`
- `PrismStreamEvent::toLaravelStreamEvent()` translates to Laravel `ToolCall` / `ToolResult`
- `Promptable::broadcastNow()` iterates all stream events and broadcasts via Reverb
- Frontend `.tool_call` / `.tool_result` listeners should match

The indicator code is correct — it only appears when the model actually calls tools (HttpRequest, Bash, CurrentDateTime). Pure text responses won't trigger it.

## Session 4: mmc Full Stack Migration — motd.com to markweb Mesh Node

### What changed

Migrated mmc (mail.motd.com) from legacy stack (Nginx + Postfix + Dovecot + MariaDB) to the standard markweb mesh stack. All 9 phases completed in one session. No code commits — this was pure infrastructure work.

### Phase-by-phase summary

1. **PostgreSQL 17 + pgvector 0.8.0** — installed from Debian 13 repos, created `markweb` database/user, password auth via 127.0.0.1
2. **FrankenPHP v1.11.2** — static binary, systemd service as `sysadm:sysadm`, Caddyfile with 3 vhosts (motd.com static, app.motd.com markweb, mail.motd.com Stalwart proxy). Reissued SSL cert to add `app.motd.com` SAN.
3. **Stalwart v0.15.5** — binary install (official installer script broken on Debian 13 dash), config modeled on mko, domain + 2 accounts (`admin@motd.com`, `ai@motd.com`), DKIM RSA key generated
4. **Ollama v0.17.0** — official installer, pulled `nomic-embed-text` (274MB F16)
5. **markweb app** — cloned via HTTPS token (stripped after), installed composer + bun deps, added missing PHP extensions (curl, dom, zip, pgsql, xml, bcmath), ran 26 migrations, built frontend on server
6. **Reverb** — systemd service using `frankenphp php-cli`, binds 127.0.0.1:8080
7. **Service swap** — stopped/disabled nginx, postfix, dovecot, opendkim, opendmarc, mariadb; started postgresql, stalwart, frankenphp, reverb. All ports verified.
8. **DNS** — DKIM record updated with Stalwart's new key on ns1gc, cleaned up bogus `mail._domainkey.motd.com.motd.com` duplicate, serial bumped
9. **Mesh heartbeat** — systemd timer every 30s POSTs to mko via WG, `/etc/hosts` entry for markweb.kanary.org → 172.16.1.210, mmc showing online on mko dashboard

### NetServa user pattern

Followed existing convention: `sysadm` (1000:1000) runs services, `u1001` (1001) owns motd.com static site, new `u1002` (1002) owns app.motd.com markweb instance.

### Gotchas discovered

- **sshd `MaxStartups 1`** — caused intermittent "Not allowed at this time" rejections. Bumped to `10:30:60`.
- **Stalwart v0.15.5 CLI** — no `account` subcommand; must use REST API (`POST /api/principal`) for account/domain management
- **Stalwart requires domain first** — creating a user principal for `admin@motd.com` fails with `notFound: motd.com` until the domain principal exists
- **FrankenPHP home dir perms** — `u1002` home dir was `700`, blocking `sysadm` from `chdir` into WorkingDirectory. Fixed to `755`.
- **curl SNI on localhost** — `curl -sk https://localhost` fails when certs don't match; need `--resolve host:443:127.0.0.1`
- **kanary.org DNS from mmc** — public DNS resolves to wrong host; added `/etc/hosts` for WG routing
- **Stalwart install script** — `curl | sh` fails on Debian 13 dash shell; direct binary download works
- **mmc RAM boost** — VM went from 2GB to 8GB; CPU usage dropped dramatically because kernel stopped thrashing page cache (kswapd idle)
