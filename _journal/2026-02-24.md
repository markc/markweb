## Session 2: Async Queue, Tool Indicator, and Docs

### What changed

Reverted to async queue dispatch (Opus 4.6 too slow for sync), fixed infrastructure, added tool call UX, and started `/docs/` collection.

**4 commits:** `3b03219..224b41e`

1. **`3b03219` — Revert to async dispatch**
   - `dispatchSync()` blocks the web request for the entire LLM response — times out on Opus 4.6
   - Reverted to `dispatch()` (async queue) — returns session_key immediately, streams via Reverb
   - Queue worker service (`markweb-queue.service`) required again

2. **`82c0f1c` — Bump ProcessChatMessage timeout to 600s**
   - Opus 4.6 with tool calls exceeds the 300s default
   - Increased to 600s (10 minutes)

3. **`88a8533` — Add tool executing indicator**
   - Listen for `.tool_call` / `.tool_result` WebSocket events from laravel/ai SDK
   - Show spinner with tool name (e.g. "Using HttpRequest...") in message bubble
   - Replaces silent stall during tool execution gaps

4. **`224b41e` — Add websocket-topologies.html to public/docs/**
   - AI-generated interactive tutorial on WebSocket topologies in Laravel Reverb

### Server infrastructure fixes (Session 2)

- Created `markweb-reverb.service` replacing `laramail-reverb.service` (was using wrong app WorkingDirectory)
- Created `markweb-queue.service` (markweb had no queue worker — laraclaw/laramail workers don't process markweb's queue)
- Added `REVERB_MAX_REQUEST_SIZE=1000000` and `REVERB_APP_MAX_MESSAGE_SIZE=1000000` to server `.env` (default 10KB too small, caused "Payload too large" errors mid-stream)
- Killed stale laraclaw/laramail queue workers running on mko

### Key insight: tool call streaming gaps

When the agent uses tools, the flow is:
```
text streams → model calls tool → SILENCE (tool executes) → tool result → text resumes
```
Each tool call is a separate LLM API round-trip. The silence can last 5-30 seconds depending on the tool. The tool indicator now makes this visible to the user.

### Gotchas discovered

- `env()` returns null after `config:cache` — never call it outside config files
- Laravel's default broadcasting config uses public-facing `REVERB_HOST`/`REVERB_PORT` for both frontend AND server-side Pusher client — breaks with reverse proxy (Caddy)
- Reverb reads app credentials at boot — restarting Reverb required after changing `REVERB_APP_ID`
- Queue workers from other apps (laraclaw/laramail) don't process markweb's database queue
- `dispatchSync()` is simpler but times out on slow models (Opus 4.6) — async queue required
- Reverb default `max_message_size` is 10KB — far too small for LLM streaming with tool results
- Systemd services for queue/reverb need restarting after config changes — `queue:restart` signal unreliable for stale workers
