## Session 1: Fix Chat Page — Send, Streaming, and Infrastructure

### What changed

Fixed the `/chat` page end-to-end: messages now send, stream, and render with markdown.

**6 commits:** `44da523..b5df807`

1. **`44da523` — Fix chat send route and add markdown rendering**
   - Replaced broken Ziggy `route('chat.send')` with Wayfinder `send.url()` import
   - Added Streamdown + `@streamdown/code` plugin (github-dark theme) for assistant messages
   - User messages stay plain text; assistant messages render markdown with syntax highlighting
   - Streaming messages use `mode="stream"` for incremental token rendering

2. **`77f6348` — Fix ModelRegistry: resolve env vars at config-cache time**
   - `config/agent.php` stored env var *names* (`'ANTHROPIC_API_KEY'`) instead of values
   - `ModelRegistry` called `env($envKey)` at runtime — returns null after `config:cache`
   - Fixed to store `env('ANTHROPIC_API_KEY')` in config (resolved at cache time)
   - This fixed the "No API keys configured" dropdown

3. **`0b54eef` — Add session lifecycle subscription for sidebar refresh**
   - Added Echo subscription to `chat.user.{userId}` in `app-dual-sidebar-layout.tsx`
   - Sidebar conversations now refresh in real-time on session create/update/delete
   - Ported from laraclaw's ChatLayout pattern

4. **`6c27d5e` — Add isSending state for spinner between send and stream start**
   - Spinner now shows immediately on send (not just after `stream_start`)
   - Covers the gap while the job is queued/processed
   - Input and button disabled during both sending and streaming

5. **`0c1d165` — Publish broadcasting config for internal Reverb connection**
   - Default Laravel config uses `REVERB_HOST`/`REVERB_PORT` (public TLS endpoint) for server-side Pusher client
   - This routed through Caddy which returned 404 for `/apps/markweb/events`
   - Published `config/broadcasting.php` pointing to `127.0.0.1:8080` (direct to Reverb)

6. **`b5df807` — Use dispatchSync, eliminate queue worker dependency**
   - Changed `ProcessChatMessage::dispatch()` to `dispatchSync()`
   - Job runs inline in web process — no queue worker service needed
   - Removes one entire systemd service from the infrastructure

### Laraclaw comparison

Thorough comparison showed markweb already had all core services identical to laraclaw:
AgentRuntime, IntentRouter, MemorySearchService, ContentSanitizer, ToolResolver, ProcessChatMessage.

markweb is actually **better** in two areas:
- Broadcast channel auth validates session ownership (not just user ID)
- Message rendering has Streamdown markdown + syntax highlighting (laraclaw is plain text)

### Gotchas discovered

- `env()` returns null after `config:cache` — never call it outside config files
- Laravel's default broadcasting config uses public-facing `REVERB_HOST`/`REVERB_PORT` for both frontend AND server-side Pusher client — breaks with reverse proxy (Caddy)
- Reverb reads app credentials at boot — restarting Reverb required after changing `REVERB_APP_ID`
- Queue workers from other apps (laraclaw/laramail) don't process markweb's database queue
- `dispatchSync()` is simpler than async queue for chat — one less service to manage
