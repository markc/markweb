### Thunderbird autoconfig DNS fix (continued from 2026-02-25)

Root cause: `autoconfig.goldcoast.org` was a CNAME to `mail.goldcoast.org` on ns1gc, and the wg0 DNS routing only covered specific subdomains (`~web.goldcoast.org`, `~mail.goldcoast.org`) — not the full zone. **Three fixes applied:**

1. **ns1gc (public DNS):** Replaced `autoconfig`, `autodiscover`, `mta-sts` CNAMEs with A records pointing to the public IP. Never use CNAMEs for records that differ between public and WG DNS.
2. **mko pdns (WG DNS):** Replaced same CNAMEs with A records pointing to WG IP (172.16.1.4). Zone transferred to cachyos secondary automatically.
3. **wg0.conf PostUp:** Changed from `~web.goldcoast.org ~mail.goldcoast.org` to `~goldcoast.org` — routes ALL goldcoast.org subdomain queries to the local pdns via systemd-resolved.

### Gotchas discovered

- **Stalwart principals need explicit permissions** — creating a principal without `enabledPermissions` gives a user that can authenticate but can't do anything. Always include the full permission set.
- **PATCH doesn't work on v0.15.5** — use DELETE + POST to update principals. The JSON deserialization fails for PATCH requests.
- **mmc admin password differs** — `WM5JwCRbG3O7` (not `changeme_N0W`). Check `config.toml` `[authentication.fallback-admin]` section.
- **`mail.<domain>` must be a full Stalwart proxy** — not a PHP app with selective JMAP path matching. CalDAV/CardDAV needs `/.well-known/*` and `/dav/*` paths proxied too.
- **DAV endpoints** — CalDAV at `/dav/cal/<email>/`, CardDAV at `/dav/card/<email>/`, WebDAV files at `/dav/file/<email>/`. Stalwart auto-creates default calendar and address book per principal.

### Filesystem blob store (cachyos)

Switched cachyos Stalwart from `blob = "rocksdb"` to `blob = "fs"` to test the filesystem blob backend.

#### Config change (`/etc/stalwart/config.toml`)

```toml
[store.fs]
type = "fs"
path = "/var/lib/stalwart/blobs"

[storage]
blob = "fs"    # was "rocksdb"
```

Created `/var/lib/stalwart/blobs/` owned by `stalwart:stalwart`. Restarted `stalwart-mail`.

#### Result: zero overhead

| File | Input size | On-disk size | Overhead |
|------|-----------|-------------|----------|
| fs-test.txt | 25 B | 25 B | 0% |
| test-100k.bin | 102,400 B | 102,400 B | 0% |

Files stored as content-addressed hashes in `XX/YY/<hash>` directory structure — not original filenames. The filename-to-blob mapping lives in the RocksDB metadata store.

RocksDB blob storage incurs ~20-30% overhead from compaction and write-ahead logs. The fs backend stores raw bytes with zero penalty, and files are accessible to standard tools (rsync, borg, filesystem snapshots/compression).

#### Other blob backends available

| Backend | Config value | Use case |
|---------|-------------|----------|
| RocksDB | `rocksdb` | Default, zero-ops, embedded |
| Filesystem | `fs` | Inspectable, rsync-able, zero overhead |
| PostgreSQL | `postgresql` | Unified backups with existing pg |
| S3-compatible | `s3` | Offload to cheap bulk storage (MinIO, AWS, B2) |
| MySQL | `mysql` | Alternative RDBMS |

Backends are configured independently per function: `data`, `blob`, `fts`, `lookup` — can mix (e.g. RocksDB for metadata + fs for blobs).

#### Capacity planning (1000 users @ 100MB quota)

- 100GB raw storage, zero overhead with fs backend
- SSD recommended but not required (WebDAV is I/O bound)
- 4-8GB RAM for RocksDB metadata cache
- Main bottleneck is concurrent DAV sync connections, not storage

#### RocksDB vs Filesystem — pros and cons

| Factor | RocksDB | Filesystem | PostgreSQL |
|--------|---------|------------|------------|
| Storage overhead | ~20-30% (compaction/WAL/SST) | **0%** (byte-for-byte) | ~30-40% (TOAST/MVCC/WAL) |
| RAM usage | 4-8GB (block cache + bloom) | Negligible (kernel page cache) | 8-16GB (shared_buffers + TOAST) |
| CPU | Compaction spikes (unpredictable) | Flat | VACUUM steady but constant |
| Write amplification | High (3-10x from compaction) | **1x** (one write, done) | Moderate (WAL + heap + TOAST + VACUUM) |
| Read latency | Sub-ms (memory-mapped SST) | Sub-ms (kernel page cache) | 1-5ms (query parse + TOAST detoast) |
| Backup | Snapshot or export tool | rsync/borg/restic directly | pg_dump / pg_basebackup |
| FS features | Opaque | btrfs/zfs compression, snapshots, dedup | Opaque |
| Compaction/VACUUM | Storms at scale, can't defer | None | Must tune autovacuum, bloat kills perf |
| Crash consistency | WAL — excellent | Depends on FS journal (ext4/btrfs fine) | WAL — excellent |
| Small blobs (<4KB) | Excellent (inline in LSM) | Wastes a full FS block per blob | Good (inline in heap row) |
| Large blobs (>2KB) | Causes LSM bloat | Excellent — no overhead | TOAST chunks, reassembly overhead |
| Max practical size | ~500GB | Disk limit (multi-TB fine) | ~2TB before VACUUM gets hard |
| Ops complexity | Zero (embedded) | Low (just a directory) | High (separate service, tuning) |

**At scale (1000 users @ 1GB = 1TB):** RocksDB needs 1.3TB disk + 8GB RAM. Filesystem needs 1.0TB + 2GB. PostgreSQL needs 1.4TB + 16GB.

**Verdict:** fs for mail servers at any scale — zero overhead, zero RAM cost, zero maintenance, standard backup tools. RocksDB for tiny deployments (<100GB) where single-dir simplicity matters. PostgreSQL only if unified pg_dump backup justifies the 40% overhead and RAM cost.

#### Migration caveat

Existing blobs written before the switch remain in RocksDB. Only new content goes to fs. Full migration requires export/re-import — not worth it for small datasets currently on these nodes.

### ZFS tuning for Stalwart blob storage

With `blob = "fs"`, ZFS can be tuned per storage function. RocksDB metadata and blobs have fundamentally different I/O patterns — separate datasets are essential.

#### Dataset layout

```bash
# RocksDB metadata (small random I/O, LSM tree SST files)
zfs create -o recordsize=16K -o compression=lz4 -o logbias=latency \
           -o atime=off -o xattr=sa -o primarycache=all zpool/stalwart/data

# Blob store (variable-size files, write-once read-many)
zfs create -o recordsize=64K -o compression=zstd -o logbias=throughput \
           -o atime=off -o xattr=sa -o primarycache=all zpool/stalwart/blobs
```

#### Property rationale

| Property | `data` (RocksDB) | `blobs` (fs) | Why |
|----------|-----------------|-------------|-----|
| recordsize | 16K | 64K | RocksDB does 4-64KB random I/O; 16K minimises read amplification. Blobs are mostly 2-100KB emails; 64K is the sweet spot (128K wastes on small emails, 16K fragments attachments) |
| compression | lz4 | zstd | RocksDB is latency-sensitive — lz4 near-zero CPU. Blobs are write-once/read-many — zstd compresses text email 3-5x |
| logbias | latency | throughput | RocksDB WAL writes are small + fsync-heavy. Blob writes are larger sequential PUTs |
| atime | off | off | Never useful for mail — saves one write per read |
| xattr | sa | sa | Store in inode, avoids extra seeks |
| dedup | **off** | **off** | Tempting but costs 5GB RAM per TB — not worth it |

#### Compression expectations (zstd on blobs)

| Content type | Compression ratio | 1TB raw → on disk |
|-------------|-------------------|-------------------|
| Plain text email | 4-5x | ~200-250GB |
| HTML email | 3-4x | ~250-330GB |
| PDF/Office attachments | 1.1-1.5x | ~700-900GB |
| Images/video | 1.0x (incompressible) | ~1TB |
| **Mixed typical mailbox** | **2-3x** | **~350-500GB** |

1000 users @ 1GB quota = 1TB raw blobs → **350-500GB actual disk** with zstd on a typical email-heavy workload. Combined with zero overhead from the fs backend, ZFS makes the 1GB-per-user quota very practical.
