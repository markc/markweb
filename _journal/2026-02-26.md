### Thunderbird autoconfig DNS fix (continued from 2026-02-25)

Root cause: `autoconfig.goldcoast.org` was a CNAME to `mail.goldcoast.org` on ns1gc, and the wg0 DNS routing only covered specific subdomains (`~web.goldcoast.org`, `~mail.goldcoast.org`) — not the full zone. Autoconfig queries went to public DNS, resolved to the public IP, and missed the local Stalwart.

**Three fixes applied:**

1. **ns1gc (public DNS):** Replaced `autoconfig`, `autodiscover`, `mta-sts` CNAMEs with A records pointing to the public IP. Never use CNAMEs for records that differ between public and WG DNS.
2. **mko pdns (WG DNS):** Replaced same CNAMEs with A records pointing to WG IP (172.16.1.4). Zone transferred to cachyos secondary automatically.
3. **wg0.conf PostUp:** Changed from `~web.goldcoast.org ~mail.goldcoast.org` to `~goldcoast.org` — routes ALL goldcoast.org subdomain queries to the local pdns via systemd-resolved.

### Gotchas discovered

- **Stalwart principals need explicit permissions** — creating a principal without `enabledPermissions` gives a user that can authenticate but can't do anything. Always include the full permission set.
- **PATCH doesn't work on v0.15.5** — use DELETE + POST to update principals. The JSON deserialization fails for PATCH requests.
- **mmc admin password differs** — `WM5JwCRbG3O7` (not `changeme_N0W`). Check `config.toml` `[authentication.fallback-admin]` section.
- **`mail.<domain>` must be a full Stalwart proxy** — not a PHP app with selective JMAP path matching. CalDAV/CardDAV needs `/.well-known/*` and `/dav/*` paths proxied too.
- **DAV endpoints** — CalDAV at `/dav/cal/<email>/`, CardDAV at `/dav/card/<email>/`, WebDAV files at `/dav/file/<email>/`. Stalwart auto-creates default calendar and address book per principal.

### Filesystem blob store (cachyos)

Switched cachyos Stalwart from `blob = "rocksdb"` to `blob = "fs"` to test the filesystem blob backend.

#### Config change (`/etc/stalwart/config.toml`)

```toml
[store.fs]
type = "fs"
path = "/var/lib/stalwart/blobs"

[storage]
blob = "fs"    # was "rocksdb"
```

Created `/var/lib/stalwart/blobs/` owned by `stalwart:stalwart`. Restarted `stalwart-mail`.

#### Result: zero overhead

| File | Input size | On-disk size | Overhead |
|------|-----------|-------------|----------|
| fs-test.txt | 25 B | 25 B | 0% |
| test-100k.bin | 102,400 B | 102,400 B | 0% |

Files stored as content-addressed hashes in `XX/YY/<hash>` directory structure — not original filenames. The filename-to-blob mapping lives in the RocksDB metadata store.

RocksDB blob storage incurs ~20-30% overhead from compaction and write-ahead logs. The fs backend stores raw bytes with zero penalty, and files are accessible to standard tools (rsync, borg, filesystem snapshots/compression).

#### Other blob backends available

| Backend | Config value | Use case |
|---------|-------------|----------|
| RocksDB | `rocksdb` | Default, zero-ops, embedded |
| Filesystem | `fs` | Inspectable, rsync-able, zero overhead |
| PostgreSQL | `postgresql` | Unified backups with existing pg |
| S3-compatible | `s3` | Offload to cheap bulk storage (MinIO, AWS, B2) |
| MySQL | `mysql` | Alternative RDBMS |

Backends are configured independently per function: `data`, `blob`, `fts`, `lookup` — can mix (e.g. RocksDB for metadata + fs for blobs).

#### Capacity planning (1000 users @ 100MB quota)

- 100GB raw storage, zero overhead with fs backend
- SSD recommended but not required (WebDAV is I/O bound)
- 4-8GB RAM for RocksDB metadata cache
- Main bottleneck is concurrent DAV sync connections, not storage

#### RocksDB vs Filesystem — pros and cons

| Factor | RocksDB | Filesystem |
|--------|---------|------------|
| Storage overhead | ~20-30% (compaction, WAL, SST) | 0% (byte-for-byte) |
| Backup | Needs consistent snapshot or export tool | rsync/borg/restic work directly |
| FS features | Opaque — no compression/dedup/snapshots | btrfs/zfs compression, snapshots, dedup |
| Compaction storms | Yes — large mailboxes trigger I/O spikes | No — direct writes, no background rewriting |
| Atomic writes | Built-in WAL crash consistency | Depends on FS journaling (ext4/btrfs fine) |
| Small file perf | Excellent — optimised for small KV pairs | May suffer inode overhead on millions of tiny blobs |
| Operational simplicity | Single data dir, nothing extra | Must manage blob dir permissions, monitor inodes |
| Portability | Copy one dir | Must copy both RocksDB data dir AND blob dir |

**Verdict:** fs is better for mail servers where email+attachments dominate storage. RocksDB is better for small deployments where operational simplicity trumps efficiency.

#### Migration caveat

Existing blobs written before the switch remain in RocksDB. Only new content goes to fs. Full migration requires export/re-import — not worth it for small datasets currently on these nodes.
