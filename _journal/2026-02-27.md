# 2026-02-27

## Session 1: gc Container Bootstrap + 3-Node Deploy

### What changed

Brought the `gc` Incus container (Debian 13) up to full mesh-ready status as a security-isolated markweb node, replacing the direct cachyos host exposure. Added gc as a deploy target and verified full 3-node deployment.

### gc container audit

Found the container was mostly provisioned — FrankenPHP 1.11.2, PHP 8.4, PostgreSQL 17 (27 migrations ran), Stalwart 0.15.5, Valkey, PowerDNS, SSL certs, frontend built. Missing: SSH, Ollama, bun, WireGuard forwarding, git repo tracking.

### DNS fix inside gc

Unbound's catch-all forward zone pointed to `10.11.0.1` (Incus bridge gateway) but no DNS listener existed there — host runs `systemd-resolved` on `127.0.0.53` only. Fixed by changing the catch-all forwarders to `1.1.1.1` / `8.8.8.8` in `/etc/unbound/unbound.conf.d/gc-resolver.conf`. Mesh zone forwarding (kanary.org, motd.com, goldcoast.org) to local PDNS was already correct.

### SSH setup

Installed `openssh-server`, enabled the service. Generated `ed25519` keypair on cachyos host (`~/.ssh/id_ed25519`), added pubkey to both `root` and `u1002` authorized_keys inside gc. Verified: `ssh root@10.11.0.2` and `ssh u1002@10.11.0.2` both work.

### Ollama inside gc (self-contained)

Installed ollama + `nomic-embed-text` model inside gc for self-contained embeddings. Updated `.env` to point `OLLAMA_URL` and `OLLAMA_HOST` to `127.0.0.1:11434` instead of the host (`10.11.0.1`). This makes gc a complete standalone node for resource benchmarking.

Cachyos host ollama retains `phi4` (14B params, 9.1GB) for local LLM testing, separate from the mesh embedding model.

### WireGuard DNAT forwarding

gc doesn't run WireGuard — the cachyos host terminates WG and forwards mesh traffic into the container via iptables DNAT. Added to `/etc/wireguard/wg0.conf` PostUp/PostDown:

- **PREROUTING DNAT:** traffic on wg0 for `172.16.1.4` ports 22,25,443,465,993,8443 → `10.11.0.2`
- **FORWARD ACCEPT:** wg0 → incusbr0 for those ports
- **POSTROUTING MASQUERADE:** incusbr0 → gc

Verified from mko: `curl -sfk https://172.16.1.4/` returns 401 (basicauth working). SSH DNAT also verified (connection reaches gc, auth rejection as expected since mko has no key for gc).

**Gotcha:** Traffic from cachyos itself to `172.16.1.4` doesn't hit PREROUTING (it's a local address routed via `lo`). Access gc directly via `10.11.0.2` from the host.

### Git repo initialisation in gc

App was deployed via copy, not git clone. Initialised git repo in place: `git init`, added HTTPS remote (no deploy keys per security policy), fetched origin/main, set upstream tracking. Clean working tree confirmed.

### Deploy script update

Added gc as a third deploy target in `bin/deploy`:
- New `NODE_SSH` associative array decouples node names from SSH hosts (gc uses `10.11.0.2`)
- Artisan cache commands now run as vhost user (`sudo -u $user`) instead of root
- `bin/deploy gc`, `bin/deploy all` both work
