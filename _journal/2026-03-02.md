### MeshNodeUpdated broadcast spam fix

**Problem:** `composer dev` showed MeshNodeUpdated events at ~2/second, heavy CPU load, 8,489 stale jobs backed up in database queue.

**Root cause (threefold):**
1. Scheduler tasks in `routes/console.php` broadcasting on every 30s tick regardless of state change
2. HTTP heartbeat endpoint in `MeshController::heartbeat()` broadcasting on every POST
3. Massive queue backlog — worker draining stale jobs, not processing live events

**Fix:**
- Added `wasChanged(['status', 'wg_ip'])` guards to all three broadcast sites (`c306f81`, `2c598cb`)
- Flushed 8,489 stale jobs: `DB::table('jobs')->where('payload', 'like', '%MeshNodeUpdated%')->delete()`

### Mesh Architecture Design

Extended session evaluating mesh architecture and the AMP protocol:

- **AMP Protocol evaluation** — assessed fitness as WebSocket wire format. Human-debuggable framing, trivial parsers, stream-friendly `---` delimiters. Concerns: inline JSON newlines, text-only (no binary), DNS addressing is aspirational.
- **WebSocket vs HTTP POST** — strong case for persistent connections (instant failure detection, no queue involvement, natural backpressure). The heartbeat spam incident proved the fragility of HTTP POST approach.
- **Three-plane architecture** — control plane (AMP over WebSocket), data plane (WebRTC for binary), identity plane (DNS-like discovery).
- **Two-tier node model** — server nodes (always-on, WireGuard, trusted) and browser nodes (ephemeral, WebSocket through nearest server, authenticated per-session). Browsers as mesh participants means node count can scale from 3 to hundreds.
- **WebRTC for binary** — server-to-server WebRTC exists (`webrtc-rs`). WebSocket for signalling, WebRTC for media — designed to work together. WebRTC-only rejected (bootstrap problem: needs WebSocket for signalling anyway).
- **Reverb vs raw WebSocket** — Reverb's Pusher protocol adds overhead for server-to-server. Raw WebSocket for mesh control, Reverb for browser-facing.
- **Three-Reader Principle** — the core design insight. Every AMP message serves machines (routing), humans (debugging), and AI agents (reasoning) simultaneously. AI agents as first-class mesh nodes, reading/writing AMP natively without tool definitions.

## Session 2: Project Naming & Domain Strategy

### NodeMesh — top-level project label

After evaluating multiple candidates for a unified project name encompassing both appmesh (desktop automation) and markweb (server mesh), settled on **NodeMesh** as the umbrella label.

**Candidates evaluated:**

| Name | Verdict | Reason |
|------|---------|--------|
| **Nexus** | Good placeholder, collision risk | GraphQL Nexus (3.4k stars), Sonatype Nexus (2.4k), Nexus Mods (2.1k) — heavily used |
| **AMP** | Protocol name, not project name | amphp/amp (4.4k stars) is a PHP concurrency framework — fatal collision in our own ecosystem |
| **Nexamp** | Phonetically perfect, legally impossible | Nexamp Inc is a billion-dollar US solar company ($3.3B raised, owns .com, .app, GitHub org) |
| **NeX** | Visually strong, practically crowded | Nex Playground (gaming hardware, 250k+ units sold), nex.dev taken, search noise from Next.js |
| **NodeMesh** | Winner | Near-zero collision. A Romanian bakery and a tiny Cardano service. Search is wide open |

**Why NodeMesh over AppMesh as umbrella:**
- "Node" is the correct abstraction — a desktop is a node, a server is a node, a browser tab is a node, an AI agent is a node
- "App" implies applications; "node" implies participants in a network
- AppMesh carries AWS App Mesh SEO residue (sunsetting Sept 2026 but years of Stack Overflow threads)
- NodeMesh naturally contains both sub-projects without either being "primary"

**Project hierarchy:**
```
NodeMesh (nodemesh.nexus)           ← the project
├── appmesh (repo)                  ← desktop node (Rust, QML, D-Bus, FFI)
├── markweb (repo)                  ← server node / "markdown web" (Laravel, WS, WG)
├── AMP                             ← the wire protocol (.amp internal WG TLD)
└── motd.nexus                      ← reference product (email-to-AI)
```

### markweb = "markdown web"

Clarified that "mark" in markweb refers to **markdown**, not the developer's first name. Markdown is the native data format throughout: AMP wire format (frontmatter + body), journal entries, documentation, and the lingua franca of LLMs. "Markdown Web" is the brand story — a web platform where markdown is the first-class format.

- **Nexus** remains a valid working label if NodeMesh doesn't stick — the `.nexus` domains work regardless

- Don't force a name — the right one emerges from usage. NodeMesh feels natural but can be revisited when external users join
- "The appmesh + markweb project" becomes simply "NodeMesh"
