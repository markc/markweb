### MeshNodeUpdated broadcast spam fix

**Problem:** `composer dev` showed MeshNodeUpdated events at ~2/second, heavy CPU load, 8,489 stale jobs backed up in database queue.

**Root cause (threefold):**
1. Scheduler tasks in `routes/console.php` broadcasting on every 30s tick regardless of state change
2. HTTP heartbeat endpoint in `MeshController::heartbeat()` broadcasting on every POST
3. Massive queue backlog — worker draining stale jobs, not processing live events

**Fix:**
- Added `wasChanged(['status', 'wg_ip'])` guards to all three broadcast sites (`c306f81`, `2c598cb`)
- Flushed 8,489 stale jobs: `DB::table('jobs')->where('payload', 'like', '%MeshNodeUpdated%')->delete()`

### Mesh Architecture Design

Extended session evaluating mesh architecture and the AMP protocol:

- **AMP Protocol evaluation** — assessed fitness as WebSocket wire format. Human-debuggable framing, trivial parsers, stream-friendly `---` delimiters. Concerns: inline JSON newlines, text-only (no binary), DNS addressing is aspirational.
- **WebSocket vs HTTP POST** — strong case for persistent connections (instant failure detection, no queue involvement, natural backpressure). The heartbeat spam incident proved the fragility of HTTP POST approach.
- **Three-plane architecture** — control plane (AMP over WebSocket), data plane (WebRTC for binary), identity plane (DNS-like discovery).
- **Two-tier node model** — server nodes (always-on, WireGuard, trusted) and browser nodes (ephemeral, WebSocket through nearest server, authenticated per-session). Browsers as mesh participants means node count can scale from 3 to hundreds.
- **WebRTC for binary** — server-to-server WebRTC exists (`webrtc-rs`). WebSocket for signalling, WebRTC for media — designed to work together. WebRTC-only rejected (bootstrap problem: needs WebSocket for signalling anyway).
- **Reverb vs raw WebSocket** — Reverb's Pusher protocol adds overhead for server-to-server. Raw WebSocket for mesh control, Reverb for browser-facing.
- **Three-Reader Principle** — the core design insight. Every AMP message serves machines (routing), humans (debugging), and AI agents (reasoning) simultaneously. AI agents as first-class mesh nodes, reading/writing AMP natively without tool definitions.
