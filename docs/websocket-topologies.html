Laravel Reverb WebSockets

WebSocket Topologies
in Laravel Reverb

Is it strictly one-to-one? Can it do hub-and-spoke? What about many-to-many mesh? Let's break down every pattern with animated diagrams and real code.

⚡ The Short Answer

Laravel Reverb is not strictly one-to-one peer-to-peer. In fact, that's the least natural pattern for it. Reverb is a centralized WebSocket server — every client connects to Reverb (the hub), and Reverb routes messages based on channel subscriptions.

Its native, default architecture is hub-and-spoke. The server sits at the center; clients are spokes. But through Laravel's channel system (public, private, and presence channels), you can absolutely achieve:

  • One-to-One — Private channels scoped to a specific user (e.g., private-user.{id})
  • Hub-and-Spoke (Broadcast) — One event, many subscribers. This is the default pattern.
  • Many-to-Many (Mesh-like) — Presence channels where every participant can send events that all others receive, creating a logical mesh — but the physical topology always routes through the Reverb server.
✅ All three patterns are supported. The physical topology is always hub-and-spoke (server-mediated), but the logical communication pattern can be one-to-one, one-to-many, or many-to-many.

🏗️ How Reverb Actually Works

Understanding the physical architecture is key to understanding why all three patterns are possible

Physical Architecture: Always Server-Mediated

🔑 Key Insight

Reverb uses the Pusher protocol. Clients never talk directly to each other — every message goes through the Reverb server. This is what makes it a hub-and-spoke physical topology. However, by using channels intelligently, you control the logical topology — who hears what.

🔌 The Three Logical Topologies

Same physical hub-and-spoke, three different logical communication patterns

🔒

1 : 1 — One-to-One (Private)

Private channel scoped to a single user
✓ NATIVE

Use a private channel targeted at a specific user. Laravel's built-in notification broadcasting (private-App.Models.User.{id}) does exactly this.

The server dispatches an event, and only the user subscribed to that private channel receives it. Think: personal notifications, direct messages, account alerts.

⚠️ Not True Peer-to-Peer

Even in 1:1, messages route through Reverb. User A's browser sends an HTTP request to Laravel → Laravel fires an event → Reverb pushes it to User B. It's server-mediated, not WebRTC-style P2P.

app/Events/DirectMessage.php 
class DirectMessage implements ShouldBroadcast
{
    use Dispatchable, InteractsWithSockets, SerializesModels;

    public function __construct(
        public string $message,
        public int $recipientId,
    ) {}

    // Only the specific user receives this
    public function broadcastOn(): Channel
    {
        return new PrivateChannel(
            'user.' . $this->recipientId
        );
    }
}
resources/js/echo.js 
// Only User B (id=5) subscribes & receives
Echo.private('user.5')
    .listen('DirectMessage', (e) => {
        console.log(e.message);
        // → "Hey, are you free tonight?"
    });
app/Http/Controllers/ChatController.php 
// User A sends a DM to User B (id=5)
broadcast(new DirectMessage(
    message: 'Hey, are you free tonight?',
    recipientId: 5,
));
📡

1 : N — Hub-and-Spoke (Broadcast)

One source, many subscribers — the default pattern
✓ DEFAULT

This is what Reverb was designed for. The server broadcasts an event to a channel, and every client subscribed to that channel receives it simultaneously.

Use public channels for open broadcasts, or private channels for authenticated groups. Think: live dashboards, sports scores, auction updates, system-wide notifications.

💡 This IS the Pusher Model

Reverb implements the Pusher protocol. The entire Pusher/Reverb model is fundamentally a pub/sub hub-and-spoke: server publishes, clients subscribe. It's one-directional by default — server → clients. Client-to-server requires "client events" on presence channels or HTTP back to Laravel.

app/Events/ScoreUpdated.php 
class ScoreUpdated implements ShouldBroadcast
{
    public function __construct(
        public string $team,
        public int $score,
    ) {}

    // Public — anyone can subscribe
    public function broadcastOn(): Channel
    {
        return new Channel('game.live');
    }
}
app/Events/TeamUpdate.php 
// Only authenticated team members receive this
public function broadcastOn(): Channel
{
    return new PrivateChannel(
        'team.' . $this->teamId
    );
}

// routes/channels.php — authorization
Broadcast::channel(
    'team.{teamId}',
    fn ($user, $teamId) =>
        $user->belongsToTeam($teamId)
);
resources/js/app.js 
// Every subscriber gets every event
Echo.channel('game.live')
    .listen('ScoreUpdated', (e) => {
        console.log(
            `${e.team} scored! Now: ${e.score}`
        );
    });
🕸️

N : N — Many-to-Many (Mesh-like)

Presence channels + client events = logical mesh
✓ VIA PATTERN

The most powerful pattern. Using presence channels, every connected user can see who's online AND send client events (whisper) that all other members receive — creating a logical many-to-many mesh.

Physically, messages still route through Reverb. But logically, any participant can send and all others receive — exactly like a mesh. Think: collaborative editing, group chat, multiplayer games, virtual classrooms.

🔥 Client Events (Whisper)

Client events bypass your Laravel backend entirely. Client A whispers → Reverb relays → Clients B, C, D receive. This is the closest to "true mesh" — no server-side event class needed. Requires presence or private channels.

resources/js/collab.js 
// Everyone in the room sees each other & communicates
Echo.join('room.42')
    .here((users) => {
        // All currently connected users
        console.log('Online:', users);
    })
    .joining((user) => {
        console.log(user.name + ' joined');
    })
    .leaving((user) => {
        console.log(user.name + ' left');
    })
    .listen('NewMessage', (e) => {
        // Server-broadcast events
        addMessage(e);
    });
resources/js/whisper.js 
// CLIENT → CLIENT via Reverb (no server roundtrip!)
// Typing indicator: A whispers, B/C/D/E all hear it

const channel = Echo.join('room.42');

// Send a client event (whisper)
channel.whisper('typing', {
    user: 'Alice',
    cursor: { x: 150, y: 320 }
});

// ALL other members receive it
channel.listenForWhisper('typing', (e) => {
    showCursor(e.user, e.cursor);
});

// Result: TRUE many-to-many communication
// Any member can whisper → all others receive
app/Events/NewMessage.php 
class NewMessage implements ShouldBroadcast
{
    public function __construct(
        public string $body,
        public User $sender,
        public int $roomId,
    ) {}

    // Presence channel — all room members receive
    public function broadcastOn(): Channel
    {
        return new PresenceChannel(
            'room.' . $this->roomId
        );
    }
}

// Any user can trigger this from their controller
// All OTHER room members receive it = mesh behavior

📺 Channel Types That Enable Each Topology

Reverb's three channel types are the building blocks for all three patterns

📢 Public Channel

No auth required. Anyone can subscribe. Best for hub-and-spoke broadcasts.

Echo.channel('news')

→ Topology: 1:N broadcast

🔐 Private Channel

Auth required via routes/channels.php. Can scope to one user or a group. Enables 1:1 and 1:N.

Echo.private('user.5')

→ Topology: 1:1 or 1:N (gated)

👥 Presence Channel

Auth + member awareness. Tracks who's online. Supports client events (whisper). Enables N:N mesh.

Echo.join('room.42')

→ Topology: N:N mesh-like

🔄 Scaling: Multi-Server Reverb

When you scale Reverb horizontally, the topology gets even more interesting

Horizontal Scaling with Redis Pub/Sub

🔄 Why This Matters for Mesh

When User A (on Server 1) whispers to a presence channel, Reverb uses Redis pub/sub to relay that client event to Servers 2 and 3, which then push it to Users C, D, E, and F. The mesh pattern works seamlessly even across multiple Reverb server instances.

🎯 Final Summary

🔒

One-to-One

Private channels scoped to individual users. Server-mediated, not true P2P. Fully supported.

📡

Hub & Spoke

The DEFAULT pattern. Reverb is literally a hub. All broadcast-based apps use this natively.

🕸️

Many-to-Many Mesh

Presence channels + client events (whisper). Logical mesh routed through Reverb. Fully supported.

💡 The Architectural Truth

Reverb's physical topology is always hub-and-spoke — it's a centralized WebSocket server (or cluster of servers synced via Redis). Clients never connect directly to each other.

But the logical topology is whatever you design it to be:

  • Scope a private channel to one user → logical 1:1
  • Broadcast to a channel with many subscribers → logical 1:N fan-out
  • Use presence channels with whisper → logical N:N mesh

This is actually an advantage over true P2P/mesh: you get the simplicity and security of centralized routing with the flexibility of any logical pattern. No NAT traversal, no STUN/TURN, no connection explosion — just channels.
