Laravel Reverb Architecture Deep-Dive

WebSocket Topologies
in Laravel Reverb

Is Reverb strictly one-to-one peer-to-peer? Hub-and-spoke? Or can it do full mesh many-to-many? The answer: all three — and here's exactly how.

⚡ The Short Answer

No, Laravel Reverb is NOT limited to one-to-one. Physically, it always uses a hub-and-spoke topology (all clients connect to the Reverb server). But logically, through Laravel's channel system, it supports 1:1 (private user channels), 1:N hub-and-spoke (broadcast to subscribers — the default), and N:N mesh-like (presence channels + client events/whisper). With horizontal scaling via Redis pub/sub, the mesh works across multiple server instances.

Physical Architecture

How Reverb Actually Works

Every WebSocket connection terminates at the Reverb server. Clients never connect directly to each other. This is fundamentally different from WebRTC-style true P2P. Reverb implements the Pusher protocol on a ReactPHP event loop.

Physical Connection Topology — Always Hub & Spoke
ℹ️ Key Architectural Truth
Physical topology is always hub-and-spoke. The Reverb server is the central hub. Every client maintains a persistent WebSocket connection to it. What varies is the logical message flow pattern — which topology the messages follow — controlled entirely by your choice of channels and events.
Logical Topologies

Three Patterns, One Server

Through channel types and event routing, Reverb supports all three communication patterns. Each uses a different combination of channel types, authorization, and event flow.

● One-to-One
🔒 Private Point-to-Point
Targeted delivery to a single specific user. Messages go Server → Reverb → One Client. Nobody else on the channel sees the event.
PrivateChannel User-scoped Notifications DMs
app/Events/DirectMessage.php
class DirectMessage implements ShouldBroadcast
{
    use Dispatchable, InteractsWithSockets, SerializesModels;

    public function __construct(
        public User $sender,
        public string $message,
        public User $recipient,
    ) {}

    // Only the recipient's private channel receives this
    public function broadcastOn(): Channel
    {
        return new PrivateChannel('user.'.$this->recipient->id);
    }
}
routes/channels.php
// Only the authenticated user matching the ID can join
Broadcast::channel('user.{id}', function (User $user, int $id) {
    return (int) $user->id === (int) $id;
});
Dispatching from anywhere in your app
// Send a DM — only User #42 will receive it
broadcast(new DirectMessage(
    sender: $currentUser,
    message: 'Hey, are you coming to the meeting?',
    recipient: User::find(42),
));
JavaScript — Laravel Echo
// Each user subscribes to their own private channel
Echo.private(`user.${userId}`)
    .listen('DirectMessage', (e) => {
        showNotification(
            `Message from ${e.sender.name}: ${e.message}`
        );
    });
✅ When to use 1:1
Direct messages, personal notifications, user-specific alerts, payment confirmations, password reset completions — anywhere only one specific user should see the event.
● Hub & Spoke (Default)
📡 One-to-Many Broadcast
The default pattern. Server broadcasts events to all subscribers on a channel. This is what Laravel's broadcasting was fundamentally designed for.
Channel PrivateChannel Server → All ShouldBroadcast
app/Events/ScoreUpdated.php
class ScoreUpdated implements ShouldBroadcast
{
    use Dispatchable, InteractsWithSockets, SerializesModels;

    public function __construct(
        public string $team,
        public int $score,
    ) {}

    // Every subscriber on this channel gets the event
    public function broadcastOn(): Channel
    {
        return new Channel('scores');  // Public — no auth needed
    }
}
Public channel — anyone can subscribe
// No channel auth needed for public channels.
// In broadcastOn():
return new Channel('scores');

// Great for: live scores, news feeds, stock tickers,
// public dashboards, site-wide announcements
Private channel — team members only
// In the Event class:
return new PrivateChannel('team.'.$this->teamId);

// In routes/channels.php:
Broadcast::channel('team.{teamId}', function ($user, $teamId) {
    return $user->belongsToTeam($teamId);
});

// All authorized team members see the broadcast
JavaScript — Laravel Echo
// Public channel — anyone
Echo.channel('scores')
    .listen('ScoreUpdated', (e) => {
        updateScoreboard(e.team, e.score);
    });

// Private channel — requires auth
Echo.private(`team.${teamId}`)
    .listen('TaskAssigned', (e) => {
        addToTaskList(e.task);
    });
📡 This is the default pattern
When the Laravel docs say "broadcasting," this is what they mean — server-originated events fanning out to all subscribers. One event dispatch, N clients receive. The server is the only message source.
● Many-to-Many Mesh
🕸️ Logical Full Mesh via Presence + Whisper
Every participant can both send and receive. Uses Presence Channels for member awareness and Client Events (whisper) for direct client-to-client-via-server communication.
PresenceChannel Client Events Whisper N:N Collaborative
routes/channels.php — Presence channel authorization
// Return user data to make it available to all members
Broadcast::channel('room.{roomId}', function ($user, $roomId) {
    if ($user->canJoinRoom($roomId)) {
        return [
            'id' => $user->id,
            'name' => $user->name,
            'avatar' => $user->avatar_url,
        ];
    }
});
JavaScript — Client Events bypass the Laravel backend entirely
// 🔥 THE KEY MECHANISM: whisper()
// Client → Reverb → All Other Clients (NO server roundtrip!)

const channel = Echo.join(`room.${roomId}`);

// ANY member can send a client event
channel.whisper('typing', {
    user: currentUser.name,
    cursor: { x: 150, y: 320 },
});

// ALL other members receive it
channel.listenForWhisper('typing', (e) => {
    showRemoteCursor(e.user, e.cursor);
});

// When EVERY participant whispers and EVERY participant
// listens → you have a logical FULL MESH
JavaScript — Member awareness (who's online)
Echo.join(`room.${roomId}`)
    // Fires once with the full member list
    .here((members) => {
        onlineUsers.value = members;
    })
    // Fires when someone joins
    .joining((member) => {
        onlineUsers.value.push(member);
        showToast(`${member.name} joined`);
    })
    // Fires when someone leaves
    .leaving((member) => {
        onlineUsers.value = onlineUsers.value
            .filter(u => u.id !== member.id);
    });
Full collaborative editing example
// A complete N:N mesh pattern: collaborative document editing

const channel = Echo.join(`document.${docId}`)
    .here((users) => renderAvatars(users))
    .joining((u) => addAvatar(u))
    .leaving((u) => removeAvatar(u));

// Every keystroke from ANY user broadcasts to ALL others
editor.on('change', (delta) => {
    channel.whisper('edit', {
        userId: me.id,
        delta: delta,
        cursor: editor.getCursorPosition(),
    });
});

// Every user receives edits from every other user
channel.listenForWhisper('edit', (e) => {
    editor.applyRemoteDelta(e.delta);
    showRemoteCursor(e.userId, e.cursor);
});

// Result: N users, each sending to (N-1) others = FULL MESH
🔮 The Mesh Secret: Client Events
Whisper events go Client → Reverb → All Other Clients, completely bypassing your Laravel backend. Reverb handles the fan-out natively. When every participant whispers and listens, the logical topology is a full mesh. This is how you build Google Docs-style collaboration, multiplayer games, or live cursors.
⚠️ Important Caveat
Client events require Presence Channels (or Private Channels with client events enabled). You must also enable client_events in your Reverb/Pusher config. These events are not persisted — if a user is offline, they miss them.
Channel Types

The Three Channel Types

Channels are the routing mechanism. Your choice of channel type determines who can subscribe, what metadata is available, and whether client events are allowed.

📢
Public Channel
new Channel('name')
Open to anyone. No authentication or authorization required.
  • No auth callback needed
  • Anyone with the channel name can subscribe
  • Server-to-client events only
  • No member list / no presence awareness
  • Ideal for public data feeds
🔐
Private Channel
new PrivateChannel('name')
Requires authentication. Server verifies each subscription attempt.
  • Auth callback in routes/channels.php
  • Returns true/false for access
  • Server-to-client events
  • Client events with configuration
  • Ideal for team/user-scoped data
👥
Presence Channel
new PresenceChannel('name')
Private + member awareness. Know who's online and enable client events.
  • Auth callback returns user data array
  • here(), joining(), leaving() callbacks
  • Full member list always available
  • Client events (whisper) built-in
  • Ideal for collaboration & chat

Scaling

Horizontal Scaling with Redis Pub/Sub

When you run multiple Reverb instances, Redis pub/sub synchronizes channel state and events across all servers. The mesh topology works seamlessly even in a multi-server cluster.

🔄 Scaling Config (from the official docs)
Enable horizontal scaling by setting Redis as the pub/sub backend. Reverb uses Redis to relay events (including client whisper events) between server instances. This is configured in config/reverb.php under the scaling key. All three topologies work identically whether you run 1 or 100 Reverb server instances.
Summary

The Architectural Truth

Laravel Reverb supports ALL THREE topologies

Physically: Always hub-and-spoke. Every client connects to Reverb. Clients never connect to each other.

Logically: Your channel and event choices create the pattern. Private channels → 1:1. Standard broadcast → 1:N. Presence channels + whisper → N:N mesh.

It is NOT limited to peer-to-peer. The default mode is actually 1:N broadcasting. Mesh is an advanced but fully supported pattern.
